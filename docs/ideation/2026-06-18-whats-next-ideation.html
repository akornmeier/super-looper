Grounding Context

Codebase Context

Super Looper is a Claude Code plugin (TypeScript/Bun, ~40 skills + ~43 agents) that enforces a full ideate → plan → work → review → learn → ship loop so the plan’s goal and the run’s learnings can’t be silently dropped. Three strategy tracks: loop autonomy (the unattended loop.sh driver around the lfg pipeline, seed → reviewable PR hands-off), learning system (TS-validated docs/solutions/ capture + reuse), and plan integrity (keep the goal sharp through multi-phase work). Four key metrics: goal fidelity, learning reuse, unattended completion rate, code-review rework.

State corrections established during this run

• The learning producer is already shipped — sl-learn exists and is wired into lfg step 10 (fires after CI green, before DONE). The corpus is still near-empty (2 learnings) because runs haven’t produced at volume yet, not because the seam is missing.
• The TS learning schema is shipped (src/solutions/schema.ts, validate.ts, emit.ts, doc-model.ts).
• The loop driver has tests (tests/loop-driver.test.ts, 771 lines).
• The “reflect / refresh / route” workflow is parked (decided 2026-06-18; revisit after ~15–20 learnings accumulate). The SL-builds-SL self-edit guard holds (direct-edits + gate, per the 2026-06-16 decision) until a pinned stable plugin exists.

The shape of the opportunity

The corpus’s only repeated lesson is “partial / local green hides real gaps — a validator covering some invariants is more dangerous than none.” The external frontier (Cursor, OpenHands, Devin) is saturating on raw coding ability; reviewers now name human review bandwidth as the binding constraint, and goal-drift is a formal research area (GD_inaction — an agent “completes” work while quietly skipping a required step). Super Looper’s differentiator — structured phase enforcement — is real but currently lives in prose the agent is trusted to follow, not machine-checked artifacts. That gap is where nearly every lens landed.

1

Structured run-record — the loop’s black box

C1
Confidence 80%Complexity: MediumFoundational

Today an unattended loop.sh run leaves behind only an exit code (0–7) and a free-text transcript log. All the structured state the driver already tracks — attempt count, timed-out vs crashed, routed-via-PR, verify mode, final green, PR URL, deferral dispositions — evaporates at DONE. Emit a machine-readable run-record (one JSON/structured artifact per run, plus per-phase entry/exit) to a stable path. This single artifact is the substrate every measurement and downstream feature needs.

Basis
direct loop.sh computes timed_out, routed_via_pr, verify_green, and attempt count but renders them only to a free-text Report block and exit codes. All four strategy metrics in STRATEGY.md are marked “to define” / measured-from-logs.
Rationale
Super Looper’s strategy names four metrics it cannot currently compute, because runs leave no structured residue. This is the lowest-effort, highest-fan-out move available: the artifact — not any single consumer — is the asset, and every other reliability/learning idea here reads from it.
Downsides
Adds a schema surface to keep stable as the loop evolves; a partial record is its own “green ≠ correct” trap, so the schema must declare what it does not capture.
2

Machine-checkable phase enforcement

C2
Confidence 60%Complexity: Medium–HighConvergence headline

The whole pitch is “the goal and the learnings can’t be dropped,” yet lfg enforces its own order through prose the agent must remember (“You MUST execute every step IN ORDER”; “GATE: STOP. Verify… Do NOT proceed if no code changes were made”). Have each phase write a structured receipt (plan-exists, diff-non-empty, review-ran, learn-seam-fired) that the driver asserts before accepting DONE — exactly how loop.sh already refuses an unverified green. A skipped phase becomes a non-zero exit, not a silent omission. Scoped start: extract the opaque lfg step-6 residual-handoff into a tested script and emit phase receipts from there.

Basis
direct lfg/SKILL.md line 7 + the step-2 GATE prose; loop.sh’s target_ci_green already proves “no unverified success.” external goal-drift research: GD_inaction (skipping a required action after an intermediate phase) is named as Super Looper’s highest-risk drift mode.
Rationale
This was the single strongest cross-frame convergence — pain, inversion, assumption-breaking, leverage, cross-domain (aviation checklists), and constraint-flipping all independently landed here. It is what turns the product’s differentiation from a claim (“we enforce the loop”) into a demonstrable property.
Downsides
Larger design surface than the others; receipts the agent itself writes can be gamed unless their content is independently checkable. Worth scoping tightly (start with step-6 + one or two receipts) rather than rewriting the pipeline at once.
3

Auto-route on failure type

C7
Confidence 70%Complexity: MediumBuilds on C1

When a run fails, loop.sh does the same dumb thing every time: reset the target to a clean base and re-launch the same lfg on the same seed, up to the cap. A deterministic failure therefore recurs deterministically until the budget is exhausted. Classify the failure instead and dispatch the matching skill — a test failure routes to sl-debug, a spec-vs-implementation mismatch routes back to re-plan, a flake gets quarantined. docs/solutions/ becomes the routing table over time.

Basis
direct loop.sh’s reconcile-and-retry resets to clean base and re-launches the same seed; the grounding leverage list explicitly names “conditional skill auto-routing (test failure → debug).” external self-healing-CI “Pipeline Doctor” pattern.
Rationale
Retries are currently sequential gambling on a deterministic outcome. Routing by failure class is the difference between three identical doomed attempts and three escalating ones — and it closes the loop the grounding calls “structurally identical” to docs/solutions/ by making the solutions corpus the routing table.
Downsides
Depends on C1 for reliable failure classification; misclassification could route a real bug to “quarantine.” Sequence it after the run-record exists.
4

Learning-as-enforcement (enforces:)

C5
Confidence 80%Complexity: LowLearning system

A learning today is passive prose that sl-learnings-researcher reads at ideate/plan/review and hopes gets applied — re-running the unreliable skill-activation lottery every time. When a learning’s lesson is mechanically checkable, let it optionally emit a guard (a test, a validate-rule, a grep assertion) that fires automatically. Add an enforces: capability to the existing TS schema. Both learnings currently in the corpus qualify as ready test cases.

Basis
direct both corpus learnings are mechanically-checkable invariants (the plugin.json version gap; git not tracking empty dirs), and src/solutions/schema.ts is the substrate to hang an enforces: field on. external Kaizen: “if it doesn’t result in a concrete change it’s not a lesson, it’s a story”; Mem0’s procedural memory maps onto docs/solutions/.
Rationale
“Learning reuse” is a named key metric, and reuse-via-“researcher reads it” is the weakest possible mechanism. Converting the highest-value learnings into guards makes reuse deterministic — institutional memory that’s wired in, not remembered.
Downsides
Only a subset of learnings are mechanically checkable, so it needs a graduation criterion; an emitted guard is code that must itself be maintained.
5

Andon cord — halt-and-escalate

C6
Confidence 82%Complexity: LowSafety

The autopilot has exactly two responses to a problem: fix it, or durably defer and proceed (write it to a residual-findings file / PR note and keep going to DONE). There is no stop the line. Borrowing Toyota’s andon cord, add a configurable threshold — a class of finding (e.g. a security-reviewer hit) or a deferral count — that halts the run with a distinct exit code instead of shipping past a known defect with a sticky note.

Basis
direct lfg/SKILL.md step-6 writes residuals to a fallback file then continues; step-9 caps flaky-test loops and proceeds — both are defer-and-proceed, with no halt path. external Toyota Production System andon cord (defect halts the line at the source).
Rationale
A pushed deferral file is the autopilot’s version of shipping a known defect with a note no one reads — the literal “green ≠ correct” failure mode. Andon gives the loop a principled stop for high-severity findings, serving goal-fidelity and review-rework directly.
Downsides
A too-aggressive threshold defeats the point of unattended runs (halts become babysitting); the threshold needs to be conservative and configurable.
6

Plugin-author dev-loop harness

C10
Confidence 80%Complexity: Low–MediumDeveloper experience

The plugin’s own AGENTS.md spends ~30 lines warning that SKILL.md and agent definitions cache at session start — so editing a skill and re-dispatching it in the same session tests pre-edit content. The documented workarounds (the skill-creator eval, or restarting the session) are heavyweight, and the tax is paid on every behavioral edit across ~40 skills. Ship a thin harness (bun run dev:skill <name>) that spawns a fresh claude --plugin-dir against the working copy so authors edit-and-test in seconds.

Basis
direct plugins/super-looper/AGENTS.md “Validating Agent and Skill Changes”: “File edits to either layer after session start do not propagate within the same session.” The fresh-process pattern already exists in loop.sh (“each run reads the current source from disk”); dev:skill does not yet exist in package.json.
Rationale
This is the binding constraint on building Super Looper itself — the most-trafficked uninstrumented workflow in the repo. Shrinking the edit→observe cycle compounds across every future skill change.
Downsides
A spawned-subprocess harness is fiddly to make ergonomic (arg passing, output capture); it eases authoring but doesn’t replace the skill-creator eval for behavioral correctness.
7

Dry-learn mode

C14
Confidence 62%Complexity: LowLearning system

Capture today only happens for work that succeeded enough to ship — but the highest-value learnings often come from work you’d never merge. Add a mode that produces only a learning, no code: seed a risky or exploratory task, let the loop plan + spike + review without merging, and the sole committed artifact is a docs/solutions/ entry capturing what the spike discovered. The loop becomes a knowledge-acquisition engine for questions too risky to actually ship.

Basis
reasoned constraint flip — “the loop’s output is a PR of working code” inverted reveals the loop’s scarcest output is learnings (2 vs 40 skills), and a spike-and-discard mode harvests exactly the ones shipping can’t. direct the autopilot-capture plan calls the CI-autofix loop “a textbook learning source that currently evaporates at DONE.”
Rationale
It directly attacks the produce/consume imbalance and populates the corpus with knowledge no shipping run can produce — complementary to the (already shipped) ship-time capture seam, not a duplicate.
Downsides
Risk of low-signal learnings padding the corpus; needs a quality bar so dry-learn output earns its place. Lower urgency than C1–C6 until the corpus is actually growing.
