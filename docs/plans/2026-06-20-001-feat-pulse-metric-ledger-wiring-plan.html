Summary

Make the loop driver's per-run records durable in a committed JSONL ledger, teach the sl-product-pulse skill to read a local ledger source, and repoint unattended_completion_rate from no data (instrumentation pending) to a real success / total rate over the pulse window. Add a git-derived, clearly labeled learning-reuse proxy. Goal-fidelity scoring stays out of scope.

Problem Frame

The loop driver already emits a structured run-record per terminal run — the hard part is done. Research confirmed the gap is a corpus and a pointer, not a pipeline, but it also surfaced two facts the brainstorm assumed away:

  • The pulse skill has no file/JSONL source path. Source resolution routes a metric name to a provider token (analytics/tracing/payments/DB) and then assumes a provider query. Reading a committed ledger and aggregating success/total is not in the skill today, so a real skill change is required — a config edit alone resolves to no data.
  • The gitignored config.local.yaml does not travel to a fresh checkout or worktree (per the plugin's own config-reading guidance). The committed ledger travels to the cloud routine's checkout; the routing config does not. That bounds where the live metric can be computed.

A past CI learning adds a constraint: a directory that exists only on the author's disk is absent on a clean checkout, and code that distinguishes "missing" from "empty" then behaves differently in CI than locally.

High-Level Technical Design

The metric's data crosses a process and machine boundary the prose can name but a picture makes obvious: records are produced ephemerally on operator machines running the loop against other repos, then promoted across the durability boundary into this repo's committed ledger, where the pulse reads them.

Key Technical Decisions

KTD1

Ledger lives at docs/run-records/ledger.jsonl in this repo

Committed here rather than in a dedicated metrics repo: it travels to the cloud routine's checkout, mirrors the existing docs/pulse-reports/ precedent, and needs no second repo wired. A dedicated metrics repo only earns its cost if loop runs across many target repos need cross-project aggregation — YAGNI for one plugin.

KTD2

Append in the operator wrapper, selecting newest loop-*.json by mtime

RUN_ID embeds the loop.sh PID and a runtime timestamp and is not exported, so the wrapper cannot reconstruct the record filename a priori — it selects the newest loop-*.json in LOG_DIR by mtime. The append runs in the operator's normal shell after the real run, never under env -i, so the self-edit guard (which scopes only the claude child) never fires.

KTD3

Add a generic JSONL-ledger source kind to the shipped pulse skill

The capability is "read a local JSONL ledger and aggregate a field over the window," not "read super-looper run-records." Keeping it product-agnostic means any pulse user can point a metric at a local ledger via config, and the super-looper-specific part (which field, which predicate) stays in the metric's own definition the pulse already re-reads from STRATEGY.md.

KTD4

success := outcome == "success"; the give-up floor is already a failure

The loop's honest terminal states — DONE-but-red, timeout, cap-exhausted, the give-up floor — all map to outcome: "failure" in the record by construction (only DONE + green verification yields success). So the metric needs no new classification and cannot overstate completion by counting a clean-exit give-up as a win.

KTD5

Cloud routing stays in the self-contained routine prompt; live metric targets the local pulse

Because the gitignored config does not travel, the weekly cloud routine cannot read pulse_metric_sources. The committed ledger does travel, so the cloud routine could compute the metric — but its routing lives in the RemoteTrigger prompt, and updating that prompt is an out-of-band operation, deferred to follow-up. This plan's live metric is scoped to the local pulse.

KTD6

Tracked .gitkeep; empty-ledger is a valid no-data state distinct from missing-path

A .gitkeep keeps docs/run-records/ present on a clean checkout. The aggregator treats an absent or empty ledger as a graceful "no data yet," not a missing-path error — the distinction the past CI learning warns must behave identically locally and in CI.

U2

Operator-side ledger append in the loop wrapper

Goal
After a real loop.sh run, append the newest run-record as one compact JSONL line to the committed ledger.
Requirements
R1, R3, R4
Dependencies
U1
Files
scripts/append-run-record.sh (new), scripts/loop.example.env (modify), tests/append-run-record.test.ts (new)
Execution noteWrite the empty-LOG_DIR guard test first — a no-op on no records is the behavior that keeps the wrapper safe to run unconditionally.
Approach, patterns, test scenarios, verification

Approach: A small helper script resolves LOG_DIR (default /tmp/super-looper/loop, or the value passed to --log-dir), selects the newest loop-*.json by mtime, compacts it to a single line, and appends to docs/run-records/ledger.jsonl. The wrapper calls it after the real run (after the final loop.sh invocation), in the operator's normal shell — outside env -i. The record is an index with no inlined seed text, so the no-PII property holds without scrubbing.

Patterns to follow: scripts/loop.sh JSON emission and LOG_DIR handling; tests/loop-driver.test.ts as the TS-harness-over-shell-script pattern (temp dirs, binary stubs).

Test scenarios:

  • Happy: a LOG_DIR with several loop-*.json of differing mtimes → only the newest is appended, as one valid compact JSON line.
  • Edge: empty LOG_DIR (no records) → exits 0, appends nothing, no error.
  • Append-only: a second invocation appends a second line; prior lines are byte-preserved.
  • Integrity: the appended line round-trips as JSON with the expected run-record keys and no fields added by the wrapper.

Verification: Running the wrapper against a fixture LOG_DIR grows the ledger by exactly one well-formed line per run; an empty dir leaves it untouched.

Add a JSONL-ledger source kind to the pulse skill

Goal

Execution noteValidate behaviorally via the skill-creator eval workflow, not bun test — plugin skill prose caches at session start, so in-session typed dispatch tests pre-edit content (per AGENTS.md).
Approach, patterns, test scenarios, verification

Approach: Extend the source-resolution prose so a metric source can name a local JSONL ledger (e.g., a ledger token plus the repo-relative path, defaulting to docs/run-records/ledger.jsonl). When resolved, the agent reads the file, filters records into the pulse window by their timestamp field, and computes the metric the strategy defines — for unattended completion rate, the share with outcome == "success". An absent or empty ledger renders no data; a malformed line is skipped, not fatal. Keep the source kind generic — "a local JSONL ledger aggregated over the window" — so it is reusable beyond this product.

Patterns to follow: the existing source-resolution block in report-template.md (pending → sourced ordering, (default source) marker); the DB-read path in SKILL.md Phase 2.1 as the nearest "read local data" precedent.

Test scenarios (skill-creator evals):

  • Populated ledger, mixed outcomes in window → renders the rate (e.g., 3/4 = 75%), not no data.
  • Absent or empty ledger → renders no data, no error.
  • Records with started_at outside the window → excluded from numerator and denominator.
  • Give-up / DONE-but-red records (outcome: "failure") → counted in denominator only.
  • A malformed JSONL line present → skipped; the rest still aggregate.

Verification: A pulse run with the ledger source configured and a seeded ledger reports a concrete rate; with no ledger, the metric line reads no data and the run completes.

Risks & Dependencies

OPEN · FLAGGED
Cloud pulse won't show the metric until the routine prompt is updated. The committed ledger travels, but the gitignored routing config does not, and the weekly routine reads git signals only. A reader expecting the cloud report to show the rate will see it absent until the follow-up prompt update lands (KTD5).
OPEN · DEFERRED TO U2
Concurrent loops in one LOG_DIR can confuse newest-by-mtime selection. If an operator runs overlapping loops sharing a log dir, the wrapper may append the wrong record. Acceptable for v1 (rare); note the limit and consider keying on the record's ended_at if it becomes real.
MITIGATED
Plugin skill caching blocks in-session validation. Behavioral changes to the pulse skill (U3, U5) cannot be validated by in-session typed-agent dispatch. Mitigation: validate via skill-creator evals or a fresh session, per AGENTS.md.
MITIGATED
New committed directory absent on clean checkout. A directory that exists only locally diverges between local and CI. Mitigation: tracked .gitkeep (U1) plus an empty-ledger-is-valid path (U3); run bun test and bun run release:validate before landing since skill prose changed.

Open Questions

  • Ledger source token shape — whether the config value is ledger:<path>, a bare path, or a token plus a path default. Resolve in U3; lean to the simplest form the pulse agent can follow, defaulting the path to docs/run-records/ledger.jsonl.
  • Record selection under concurrency — newest-by-mtime versus matching the record's own ended_at. Resolve in U2; mtime for v1.
