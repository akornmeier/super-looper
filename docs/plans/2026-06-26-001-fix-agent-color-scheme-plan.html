Summary

Assign a valid color to all 43 agents by role, spreading the 8-value palette so each co-dispatched fleet's task-list panel shows several colors instead of one. Work proceeds family-by-family across four units: code-review reviewers (split into three attention tiers), the seven doc-review personas (each distinct), the researcher/auditor cluster (distinct within the plan and compound waves), and the design/writer agents. This fixes the invalid violet, the 25 unset agents, and the blue×13 collision.

Problem Frame

Of 43 agents, 25 set no color and 13 are all blue; sl-design-iterator uses violet, which the field does not accept. The collision lands where parallelism is highest — sl-code-review dispatches up to ~14 agents, most of them the blue ones — so the densest panel is nearly monochrome and color carries no signal exactly when a user is trying to tell parallel agents apart. The field accepts only eight named values, so the goal is per-panel legibility, not global per-agent uniqueness (see origin: docs/brainstorms/2026-06-26-agent-color-scheme-requirements.md).

Requirements

Carried from the origin requirements doc.

Coverage and correctness
  • R1. Every agent sets color to one of the eight valid values (red, blue, green, yellow, purple, orange, pink, cyan); none left unset.
  • R2. No off-palette value; the current violet on sl-design-iterator is corrected.
  • R3. The blue×13 collision is eliminated — no single color applied to a majority of agents as an unconsidered default.
Assignment scheme
  • R4. An agent's color is determined by its role, grouping into ~6 families (code-review, doc-review, researchers, design/UI, writers/resolvers, data/deploy).
  • R5. The palette is spread so each co-dispatched fleet's panel shows multiple colors — optimized for in-panel legibility, not global per-agent uniqueness.
  • R6. A color may carry different roles in different panels; reuse across fleets that never co-run is intended.
Per-fleet distinctness
  • R7. In a fleet of ≤8 co-dispatched agents (e.g. sl-doc-review's 7), every agent has a distinct color.
  • R8. In a fleet exceeding 8 (sl-code-review), agents are sub-split into three attention tiers — high-stakes, structural, routine — so the panel shows several colors; agents sharing a tier may share a color, disambiguated by name.
  • R9. An agent in multiple fleets keeps one fixed color, chosen to stay distinct within every fleet it joins.
  • R10. A straddling agent's color follows the fleet it dispatches in, not its topic.

Key Technical Decisions

Hand-author the colors; do not build a generator

Each color is written directly into the agent's frontmatter. The assignment changes only when agents are added or fleets reshuffle (rare), so a graph-coloring generator is carrying cost the change doesn't justify. The deferred validation test — not a generator — is what guards drift.

Code-review is split into three attention tiers

High-stakes (block-the-ship: correctness, security, adversarial) takes the loudest hue; structural and routine take their own. This folds urgency into the palette so a critical finding never renders in recessive blue, and gives the densest panel 3+ colors.

Distinctness is verified per fleet, not globally

The coloring was solved against the actual co-dispatch map: sl-doc-review (7), sl-code-review (≤16), sl-plan (14, dispatched in waves), sl-compound (8), sl-ideate grounding (4), sl-optimize (2). An agent in several fleets (e.g. sl-learnings-researcher, in 4) holds one color picked to stay distinct in each. sl-resolve-pr-feedback lists all 42 agents as a persona catalog, not a 42-way parallel panel, so it imposes no all-vs-all constraint.

The name is the identity channel

Same-color agents within a tier are told apart by the agent name, which always renders beside the chip. Color groups; the name identifies.

Scope Boundaries

Deferred to Follow-Up Work (separate PRs; recorded in the origin doc):

  • A validation test enforcing on-palette values and full color coverage across all agents — the guardrail that keeps the scheme from drifting back. Clone the pattern of tests/skill-agent-sl-prefix.test.ts.
  • Color-vision-deficiency and light/dark theme legibility rules (avoiding confusable pairs within a fleet; keeping blue/yellow off high-attention roles).
  • A deterministic generator deriving each color from a declared family/tier.

Outside this change: recoloring non-agent surfaces (status line, pulse reports, demo reels); loop-phase coloring (rejected upstream in favor of role).

Risks & Dependencies

  • · The eight-value palette is a Claude Code platform contract, not validated in this repo. If the platform later adds values (e.g. hex), the scheme still holds; this plan does not depend on that.
  • · Fleet composition can change as agents are added, which would require re-checking within-fleet distinctness. Without the deferred validation test, that check is manual until the follow-up lands.
  • · release:validate does not inspect color, so it neither blocks nor verifies this change; verification is by grep + visual panel inspection until the validation test ships.
