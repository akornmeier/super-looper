Grounding Context

The point that reshapes the whole request: Claude Code's agent color field accepts exactly eight named values — red, blue, green, yellow, purple, orange, pink, cyan. No hex, no custom values (official sub-agents docs). So 43 unique colors is literally impossible — "unique" has to be reframed before anything else is decided.

  • The current state is collision, not uniqueness. Of 43 agents, only 18 set a color and 25 set none. Of the 18: blue ×13 plus one each of cyan / purple / red / yellow / violet. Thirteen agents look identical.
  • One value is already invalid. sl-design-iterator uses violet, which is not in the eight — it ships broken and nothing caught it.
  • Nothing validates color. No reference to the field in src/release/, scripts/release/, or tests/. The only enforced frontmatter convention is the sl- name prefix (tests/skill-agent-sl-prefix.test.ts).
  • Agents run in parallel fleets — that's where color earns its keep. sl-doc-review dispatches exactly 7 personas (fits the 8 cleanly); sl-code-review dispatches 6 always-on plus conditionals (9–14, the one fleet that overflows 8). sl-learnings-researcher alone appears in 4 different fleets.
  • The name is always next to the chip. In the task list and transcript, the color renders beside the agent's (already-unique) name — so color is a fast grouping cue, not the unique identifier. Redundant encoding already exists for free.
  • Accessibility is real here. Named ANSI remaps to each user's terminal theme. blue is the classic casualty on dark themes; bright yellow washes out on light. red/green and blue/purple are the two color-vision-deficiency (CVD) confusable pairs. The 8 map closely onto the CVD-safe Okabe–Ito set.

Topic Axes

  • Assignment scheme — how the 8 colors map onto the 43 agents.
  • Accessibility & contrast — CVD-safety, light/dark legibility, which pairs to avoid.
  • Coverage & correctness — the 25 unset agents, the invalid violet, the blue×13 collision.
  • Governance & drift — keeping every future agent on-palette without manual policing.
  • Meaning of "unique" — what distinctness can mean when 43 items exceed 8 colors.

Ranked Ideas

Jump to:
  1. Color by role-family, not by agent
  2. Redefine "unique" as distinct-within-a-fleet
  3. Add a palette + coverage validation test
  4. Reserve the loud colors for block-the-ship roles
  5. Make "accessible" a checkable rule, not a vibe
  6. Make the scheme self-maintaining
1

Color by role-family, not by agent

assignment-scheme Confidence 90% Complexity Medium

Stop trying to color 43 agents and color the ~6 families they already fall into; each agent inherits its family's hue. The impossible 43→8 mapping becomes a comfortable 6→8 one, and the chip starts meaning something — you read an agent's role off its color instead of memorizing 43 arbitrary assignments. A concrete starting map (colors are illustrative of the named ANSI values):

Code-reviewcorrectness, maintainability, testing, security, performance… (~18)
Doc-reviewcoherence, feasibility, product-lens, scope-guardian… (7)
Researchersweb, repo-research, learnings, framework-docs, git-history… (~10)
Design / UIdesign-iterator, figma-design-sync, design-implementation… (~4)
Writers / resolversankane-readme-writer, pr-comment-resolver
Data / deploydata-integrity-guardian, data-migration, deployment-verification

Six families, eight colors — with red and yellow held in reserve (see idea 4). The split a reader needs is "which kind of agent is this," and the chip answers it pre-attentively.

Design fork worth deciding Color can key on role-family (above) or on loop phase (ideate / plan / work / review / learn / ship). Phase ties color to Super Looper's core narrative — a glance tells you which phase is live — but families map more cleanly onto how agents are actually grouped today. Pick one axis of meaning; don't blend both.
Basis external: mature design systems (Adobe Spectrum, Workday Canvas) and process-coloring tools (docker-compose, Turborepo) all assign color by function class, never per item, precisely because items outnumber colors. direct: the families are already visible in the agent filenames and in how skills dispatch them.
Rationale This is the one decision everything else hangs off. It converts an unsolvable constraint into a solvable one and makes color informative rather than decorative — the difference between "agent #4" and "a reviewer is running."
Downsides Agents within a family share a color, so color alone won't tell two reviewers apart (the name does). Family boundaries are a judgment call — a few agents straddle two families and need an arbitration rule.
2

Redefine "unique" as distinct-within-a-fleet

unique-meaning Confidence 88% Complexity Low

Two agents only ever compete for your eye when they're dispatched together. So "unique" doesn't mean unique across all 43 — it means no two agents that co-run share a color. That target is reachable: most fleets fit inside 8.

sl-doc-review — 7 personas, all distinct (fits 8)
adversarial-doc coherence design-lens feasibility product-lens scope-guardian security-lens
Provably collision-free — 7 personas, 8 colors.
sl-code-review — 9–14 agents (the one fleet that overflows 8)
correctness maintainability testing security performance reliability api-contract data-migration
Colors repeat here — accept it. The name disambiguates the repeats, like the two black suits in a deck.

The subtlety that makes this a real constraint, not a slogan: one agent holds one color but joins several fleets (sl-learnings-researcher is in 4), so it's a single graph-coloring of the co-dispatch graph, not independent per-fleet assignment.

Basis direct: verified fleet sizes — doc-review = 7, code-review = 6 always-on + conditionals, learnings-researcher in 4 fleets. external: transit maps reuse ~8 line colors across cities because two cities' lines never share one map; heraldry's rule of tincture and the four-color theorem are the same adjacency-coloring idea.
Rationale This is the reframe that makes the user's literal "unique" request achievable where it actually matters (inside a panel) and honestly impossible only in the single fleet that overflows — which then gets special handling instead of pretending.
Downsides Requires knowing each skill's dispatch set to assign correctly; that set changes as agents are added (which is exactly what idea 6's guardrail is for). Code-review's overflow has no clean answer — it leans on the name as the tiebreaker.
3

Add a palette + coverage validation test

coverage-correctness Confidence 92% Complexity Low

Clone the pattern the repo already trusts. A small test beside tests/skill-agent-sl-prefix.test.ts that asserts every agent has a color and that it's one of the eight valid values. It fails the build today on violet and on the 25 unset agents, then catches every future regression. Wire it into release:validate.

Basis direct: color is validated nowhere; violet is invalid and shipped; 25 agents are unset; and the sl--prefix test is a working template for exactly this kind of frontmatter contract.
Rationale Highest return for the least code. It's the difference between a one-time cleanup that decays back to blue×13 and a property that stays true forever. The violet bug is the ready-made regression-test seed.
Downsides A bare "must be on-palette" check enforces validity but not good assignment — it won't notice a fleet drifting toward one color. Pairing it with idea 2's co-dispatch check (idea 6) closes that gap.
4

Reserve the loud colors for block-the-ship roles

assignment-scheme Confidence 85% Complexity Low

Don't spread the palette evenly — weight it by attention. The perceptually loud hues go to agents whose findings can stop a merge; the recessive hues go to background workers you don't need to track moment-to-moment. Color then carries a third free signal: urgency.

LOUD — must not miss
adversarial, security, correctness, data-integrity
MID — structural
maintainability, reliability, api-contract
RECESSIVE — background
web, repo-research, learnings, framework-docs

sl-adversarial-reviewer is already red — this just makes that deliberate and pulls security off recessive blue, where a critical finding currently hides.

Basis external: design systems reserve red/yellow for high-attention roles and blue/cyan for background; biology does the same with aposematic warning coloration. direct: security currently sits on blue — both the most-collided and least-legible-on-dark color.
Rationale It spends the scarce, attention-grabbing colors where a missed signal is expensive, and it fixes the specific wrongness that today's highest-stakes reviewers render in the quietest, hardest-to-see color.
Downsides Urgency and family can conflict (a security doc-lens vs a security code-reviewer) — you need a rule for which axis wins. Over-using red dilutes its "stop" meaning.
5

Make "accessible" a checkable rule, not a vibe

accessibility-contrast Confidence 80% Complexity Low

"Readable" was half the literal ask. Turn it into two rules a validator (idea 3/6) can enforce, instead of a judgment call. Rule one: never place a CVD-confusable pair on two agents that co-dispatch. Rule two: keep blue and bright yellow off the highest-attention agents, since each vanishes on one of the two common themes.

confusable pair
red  &  green — protan/deutan CVD
confusable pair
blue  &  purple — merge for many viewers
blue on a dark theme — sinks into the background
bright yellow on a light theme — washes out

These two demos are the failure modes literally rendered — and blue is today's most-used color, on the fleet that most needs to be seen.

Basis external: the 8 map closely onto the CVD-safe Okabe–Ito palette; red/green and blue/purple are the documented confusable pairs; named ANSI remaps to the user's theme, so blue-on-dark and yellow-on-light are well-known casualties. Modern terminals add min-contrast enforcement, so violations degrade rather than disappear.
Rationale It converts "accessible" from an aspiration into a property you can test on every release — the only way it stays true as a distributed plugin ships into terminal themes you don't control.
Downsides Constraints shrink the usable palette inside a fleet, tightening an already-tight 8. Full CVD-safety for every fleet may be unreachable for the code-review overflow — treat it as best-effort there.
6

Make the scheme self-maintaining

governance-drift Confidence 70% Complexity Medium

The leverage play: spend a little structure now so the scheme survives every future agent without anyone re-deciding it. Four small moves that compound — declare the family→color taxonomy once in CONCEPTS.md (the repo's glossary home); let a metadata step derive each color from family so a new agent is colored for free; emit a color-coverage report at release time (counts per color, unset list, any fleet over its collision budget); and capture the rationale as a docs/solutions/ learning via sl-compound so the next maintainer inherits why, not just what.

Basis direct: CONCEPTS.md, extract-metadata.py-style generation, the docs/solutions/ convention, and sl-compound all already exist — this reuses the repo's own muscle rather than inventing machinery.
Rationale Without it, even a perfect one-time recoloring decays back toward blue the moment the next agent is copied from an existing one. This is what turns "a cleanup" into "a convention."
Downsides More surface than the core fix — easy to over-build. Generating color into frontmatter adds a sync step that itself needs validation. Best sequenced after ideas 1–3 land, not before.
